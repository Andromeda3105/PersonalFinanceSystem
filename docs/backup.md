\# Стратегія резервного копіювання та відновлення (Backup \& Restore)



Цей документ визначає політики збереження даних для ІС обліку персональних фінансів.



\## 1. Стратегія та Типи резервних копій

\* \*\*Повні (Full):\*\* Повний дамп усієї бази даних PostgreSQL та всіх конфігураційних файлів.

\* \*\*Інкрементальні (Incremental):\*\* Збереження файлів транзакцій (WAL - Write-Ahead Logs) бази даних для фіксації змін між повними бекапами.



\## 2. Частота створення та Ротація

\* \*\*База даних (Full):\*\* Щодня о 03:00 ночі.

\* \*\*База даних (Incremental):\*\* Щогодини.

\* \*\*Файли конфігурацій (`appsettings.json`, конфіги Nginx):\*\* При кожному деплої або зміні налаштувань.

\* \*\*Зберігання та ротація:\*\* Локальні копії зберігаються 7 днів. Хмарні копії (наприклад, в AWS S3) зберігаються 30 днів. Старіші копії видаляються автоматично.



\## 3. Процедура резервного копіювання

Процес повністю автоматизований за допомогою системного планувальника `cron` та bash-скриптів.

\* \*\*База даних:\*\* Використовується утиліта `pg\_dump`:

&#x20; `pg\_dump -U postgres -F c finance\_prod > /backups/db/finance\_$(date +%Y%m%d).dump`

\* \*\*Файли конфігурацій:\*\*

&#x20; `tar -czf /backups/config/conf\_$(date +%Y%m%d).tar.gz /var/www/finance-api/appsettings.Production.json /etc/nginx/sites-available/finance`

\* \*\*Логи системи:\*\* Логи архівуються щотижня за допомогою утиліти `logrotate`.



\## 4. Перевірка цілісності

Кожного понеділка автоматичний скрипт виконує тестове відновлення дампу в тимчасову базу даних та перевіряє структуру таблиць командою:

`pg\_restore --list /backups/db/finance\_latest.dump`



\## 5. Процедура відновлення з резервних копій

\*\*Повне відновлення системи (Disaster Recovery):\*\*

1\. Зупинка застосунку: `sudo systemctl stop finance-api`

2\. Видалення пошкодженої БД: `dropdb -U postgres finance\_prod`

3\. Створення нової БД: `createdb -U postgres finance\_prod`

4\. Відновлення з останнього повного бекапу: 

&#x20;  `pg\_restore -U postgres -d finance\_prod -1 /backups/db/finance\_latest.dump`

5\. Відновлення конфігурацій (якщо пошкоджені): розпакування архіву `.tar.gz`.

6\. Запуск застосунку: `sudo systemctl start finance-api`



\*\*Вибіркове відновлення:\*\*

У разі втрати конкретної таблиці (наприклад, `Transactions`), бекап розгортається на локальному сервері розробника, звідки експортуються лише потрібні записи у форматі CSV/SQL для ручного перенесення на production.
